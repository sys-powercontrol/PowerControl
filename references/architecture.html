Versão 1.0 - PWA/Serverless

Elaborado com base nos requisitos do PRD para garantir um ecossistema Multi-Tenant, estruturado em topologia Serverless (BaaS) e projetado com foco absoluto na resiliência operacional através da arquitetura Offline-First (PWA).

1 Visão Geral (C4 Model - Context)

O PowerControl atua como a espinha dorsal de micro e médias operações de varejo. Ele isola as complexidades fiscais e financeiras num front-end web reativo que se acopla ao Firebase (Backend-as-a-Service) e se intercomunica com Provedores Fiscais (WebmaniaBR/FocusNFe), ViaCEP e Webhooks para suporte.

2 Componentes e Topologia (PWA & Estado)

A aplicação não depende de um backend intermediário (Node.js/Python) na nuvem. Toda a regra de negócio segura reside nas integrações diretas Client -> BaaS devidamente bloqueadas por Security Rules. O estado remoto é perfeitamente espelhado com o TanStack Query.

Camada de Apresentação (UI)

  • React 18 com Vite e TypeScript.
  • Tailwind CSS e Lucide Icons (Design System).
  • React Router DOM para navegação client-side.
  • Sonner para notificações tostadas.

Camada de Estado & Cache

  • TanStack Query para cache e sincronia com backend.
  • Zustand/Context API para estado global volátil (Auth, Caixas).
  • IndexedDB (idb) para base estruturada offline.

3 Mecanismo Transversal Offline-First

Este é o diferencial de alta tolerância a falhas do ERP. O Service Worker monitora os disparos das requisições. Se houver falha de rede ou timeout na camada do Firebase SDK, o erro é contido pelo wrapper da API que salva a "Payload" na fila respectiva do IndexedDB.

Pipeline de Sincronização Assíncrona

  1. A API Mutator tenta enviar (ex: Venda, Cliente, Compra de Estoque).
  2. Em erro ou estado explicitly !navigator.onLine, aciona a offlineStore.ts.
  3. Guarda no IndexedDB e emite sync.register('sync-[dominio]') para a Thread principal do SO/Navegador.
  4. Quando a conexão é restabelecida, o Service Worker dispara o evento sync em Background.
  5. Itera sobre os itens pendentes do IDB, aplica contra o Banco de dados real de forma idêntica e expurga o item local (Max retries: 3).

4 Design de Integridade Isolada (Transactions & Locks)

Para evitar concorrência e corrupção contábil entre Caixas físicos, Contas a Pagar/Receber e o Inventário, a arquitetura lança mão de Firestore Transactions na camada da API (finance.ts, inventory.ts).

ACID Compliance Simulado

Ao registrar uma venda, o estoque e a conta bancária são atualizados numa única runTransaction. Se o caixa cair ou o estoque esgotar milissegundos antes, tudo sofre rollback.

Anti-Orphaned Writes

Estornos bancários atirando exceções caso a fonte pagadora "desapareça" do DB. Bloqueadores explícitos de reversão para fluxos atribuídos a "Turnos de Caixa Fechados".

5 Segurança Multi-Tenant (Firebase Rules)

Toda entidade é amarrada por company_id. Não dependemos do Frontend para não vazar dados. O Firebase Security Rules implementa a lógica Attribute-Based Access Control (ABAC): 

// Pseudocódigo da Arquitetura do backend Security Rule

function belongsToTenant() {
    return resource.data.company_id == request.auth.token.company_id;
}

function hasPermission(module) {
    let tenant = get(/databases/$(database)/documents/companies/$(request.auth.token.company_id));
    return tenant.data.role_permissions[request.auth.token.role].hasAny([module]);
}

match /sales/{sale} {
    allow read, write: if belongsToTenant() && hasPermission('sales.manage');
}
